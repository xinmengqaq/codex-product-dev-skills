# 交给 writing-plans 的约束

BDD 行为定向完成后，把下面约束交给 `superpowers:writing-plans`。不要在本技能里提前做工程设计。

## 标准交接块

```markdown
## 给 superpowers:writing-plans 的约束

后续工程实施计划必须以本 BDD 行为定向为唯一产品/玩法范围来源。

### 必须遵守
- 每个工程 Task 必须映射到一个 BDD Task 或 BDD 场景。
- 工程 Task 可以拆文件、接口、测试和命令，但不能改变 BDD 行为目标。
- UI、交互、状态流和体验目标必须遵守 BDD 中的 `必须遵守`。
- 实现细节只能在 BDD 的 `允许变通` 范围内自行决策。
- 每个 Task 的完成标准必须包含 BDD 中列出的真实验收证据。

### 禁止
- 不得把 BDD 标记为 `保守推断` 的内容升级成第一版必做，除非用户确认。
- 不得加入 BDD `禁止新增` 中列出的功能。
- 不得新增 UI 原型、设计文档、当前代码事实或用户明确要求没有支持的产品/玩法功能。
- 不得把纯技术动作当成最终 Task；技术步骤必须服务于对应行为闭环。
- 不得只用 mock、fixture、静态截图或代码阅读证明用户/玩家路径完成。

### writing-plans 负责
- 文件结构、模块边界、接口契约、状态设计。
- 测试代码、命令、执行顺序、提交粒度。
- 根据代码库现有模式选择实现方案。

### 本 BDD 不负责
- 不指定具体文件怎么拆。
- 不指定接口字段或数据库 schema。
- 不写测试代码。
- 不写实现步骤或提交命令。
```

## 自检问题

交接前确认：

- BDD 是否足够约束 `writing-plans`，让它不能自由脑补产品功能。
- BDD 是否仍给 `writing-plans` 留出工程判断空间。
- 是否每个 Task 都能追溯到 UI 原型、设计文档、代码事实或用户明确要求。
- 是否已经明确写出哪些东西不做。

# BDD 计划前行为定向

## 结果

本技能只做 `superpowers:writing-plans` 之前的行为门禁。

你的目标不是写工程计划、设计文件结构或实现方案，而是先回答：

- 这个产品或游戏 Task 到底要让用户/玩家完成什么行为。
- 哪些行为来自 UI 原型、设计文档、当前代码事实或用户明确要求。
- 哪些只是合理推断，必须保守处理。
- 哪些是 AI 容易臆想出的扩展功能，必须禁止进入第一版。
- 每个 Task 需要什么真实验收证据，才能交给 `superpowers:writing-plans` 继续写工程实施计划。

## 适用场景

当用户准备让 AI 开始产品、工具、网站、应用、游戏或 AI 功能开发，并且已经有 UI 原型、设计文档、规格、现有代码事实或明确目标时使用。

尤其适合这些情况：

- 用户担心 AI 在写计划时自动脑补新功能。
- 需要先把 UI 原型和设计文档转成每个 Task 的行为方向。
- 后续要调用 `superpowers:writing-plans` 写工程实施计划。
- 产品或游戏功能需要真实浏览器、试玩、真实模型调用、真实数据链路等证据。
- 用户是单人开发，需要 AI 在开工前先锁住行为闭环和边界。

不适合这些情况：

- 用户已经要求直接执行一个明确的小修复。
- 用户要的是代码实现、文件拆分、接口设计或测试代码细节。这些交给 `superpowers:writing-plans`。
- 用户只想讨论 BDD 理论或 Cucumber 语法。

## 必读导航

`SKILL.md` 只放核心流程。需要输出具体模板时读取：

- 每个 Task 的 BDD 块格式：`references/task-bdd-template.md`
- 交给 `superpowers:writing-plans` 的边界和提示：`references/writing-plans-handoff.md`

只读取当前需要的 reference。不要一次性展开无关材料。

## 真源顺序

冲突时按以下顺序判断：

1. 当前用户消息里的明确要求、禁止项和范围。
2. 用户点名的 UI 原型、设计文档、规格、计划、截图、状态卡或代码。
3. 当前代码库的真实行为和现有模式。
4. 旧会话记忆或 Graphify 等索引信息。

旧记忆和索引只能作为定位线索，不能替代当前 UI 原型、设计文档、代码和用户消息。

## 工作流程

### 1. 定位输入依据

先列出本次 BDD 的依据来源：

- UI 原型、截图、设计稿或页面文件。
- 设计文档、产品文档、游戏规则、规格或计划。
- 当前代码事实，例如已有页面、接口、状态模型、数据结构、路由、测试。
- 用户本轮明确要求。

如果用户没有给出路径，但当前仓库里存在明显的 `UI原型`、`docs`、`specs`、`plans`、`design`、`prototype` 等目录，先低成本定位。找不到再说明缺口，不要凭空补。

### 2. 分级需求来源

把需求分成四类：

- 明确依据：用户、原型、设计文档或当前代码直接支持。
- 保守推断：从明确依据能推出来，但不能扩展成新功能。
- 待确认问题：缺少关键外部信息，且选择会影响产品行为或范围。
- 禁止臆想：没有来源、只是 AI 觉得合理、会扩大第一版范围的功能。

禁止把“保守推断”写成“必须做”。禁止把“禁止臆想”放进后续计划。

### 3. 生成行为闭环

每个 Task 必须先有行为闭环，再允许进入工程计划。

行为闭环至少包含：

- 触发：用户/玩家做了什么。
- 系统反应：界面、数据、规则、模型或状态如何变化。
- 反馈：用户/玩家看见、听见或感受到什么。
- 状态变化：哪些状态必须真实改变并可验证。
- 下一步：用户/玩家自然能继续做什么。
- 失败态：失败、空输入、冲突、重复操作、模型失败或网络失败时如何恢复。

如果一个 Task 只能描述成“建表”“写接口”“做组件”“接状态管理”，说明它还不是 BDD Task；必须改写成用户/玩家可感知的行为。

### 4. 给每个 Task 写 BDD 定向块

每个 Task 前都输出一个短 BDD 块。需要模板时读取 `references/task-bdd-template.md`。

BDD 块必须覆盖：

- 输入依据。
- 行为目标。
- 正常路径。
- 失败路径。
- 边界路径。
- 必须遵守。
- 允许变通。
- 禁止新增。
- 验收证据。

BDD 块不写文件路径拆分、接口字段设计、测试代码、具体实现步骤或提交命令。这些属于 `superpowers:writing-plans`。

### 5. 生成 writing-plans 交接约束

BDD 完成后，输出一段给 `superpowers:writing-plans` 使用的交接约束。需要模板时读取 `references/writing-plans-handoff.md`。

交接约束必须说明：

- `writing-plans` 只能基于 BDD 明确依据和行为闭环写工程计划。
- 每个工程 Task 必须映射到一个 BDD Task 或 BDD 场景。
- 不得新增 BDD 没有授权的产品、玩法、AI、数据或 UI 功能。
- 没有真实验收证据的 Task 不允许算完成。
- 工程设计、文件结构、测试代码和命令由 `writing-plans` 负责，不在本技能里提前设计。

## 输出格式

默认输出：

```markdown
# BDD 行为定向

## 输入依据

## 需求来源分级

## 第一版行为闭环

## Task BDD 块

## 交给 writing-plans 的约束
```

如果用户只要求某一个 Task，则只输出该 Task 的 BDD 块和交接约束。

## 质量门

完成前自检：

- 是否所有 Task 都先有用户/玩家行为目标，而不是技术动作。
- 是否明确区分了明确依据、保守推断、待确认问题和禁止臆想。
- 是否禁止了没有来源的扩展功能。
- 是否每个 Task 都有真实验收证据。
- 是否没有提前写工程设计、文件拆分、接口字段、测试代码或命令。
- 是否能直接交给 `superpowers:writing-plans` 继续写工程实施计划。

命中以下情况必须返工：

- 把 UI 原型或设计文档没有支持的功能写进必须做。
- 把 Task 写成纯工程动作，没有行为闭环。
- 只有正常路径，没有失败路径或边界路径。
- 验收证据只写“测试通过”或“检查代码”，没有真实用户/玩家路径。
- 提前替 `writing-plans` 做工程设计。

## 停止条件

只有以下情况可以停止：

- BDD 行为定向和 `writing-plans` 交接约束已经完成。
- 缺少关键输入，且无法从当前仓库、用户消息或代码事实推断，会影响产品行为或范围。
- 用户要求停止或改方向。

停止时必须说明缺少什么、为什么它影响行为定向，以及用户最小需要补充什么。
